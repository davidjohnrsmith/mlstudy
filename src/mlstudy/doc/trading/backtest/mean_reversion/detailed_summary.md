# Detailed Summary: `src/mlstudy/trading/backtest/mean_reversion`

This document provides a detailed summary of the `src/mlstudy/trading/backtest/mean_reversion` module, which implements a sophisticated mean-reversion backtesting framework.

## 1. Overall Architecture

The module is designed to be highly modular and configurable, allowing for both single backtest runs and large-scale parameter sweeps. The main components are:

-   **`single_backtest`**: The core backtesting engine that simulates the trading strategy on a bar-by-bar basis.
-   **`sweep`**: Functionality to run multiple backtest scenarios in parallel to explore the parameter space.
-   **`configs`**: Handles the configuration of backtests and sweeps using YAML files.
-   **`data`**: Responsible for loading and preparing market data from parquet files.
-   **`parameters`**: Defines the parameters of the backtesting model.
-   **`analysis.py`**: A suite of tools for post-hoc analysis of backtest results.
-   **`plots` (in `single_backtest` and `sweep`)**: Provides visualization tools for equity curves, trade PnL, and other metrics.

## 2. `single_backtest` Module

This is the heart of the backtester.

-   **`engine.py`**: The main entry point for a single backtest run. It validates the input data and configuration, selects the appropriate loop implementation (Python or Numba JIT), and wraps the raw results in a `MRBacktestResults` object.
-   **`loop.py`**: Contains the core backtesting logic in the `_mr_loop_jit_impl` function. This function is designed to be compiled with Numba for performance. It implements a state machine with the following states: `FLAT`, `LONG`, `SHORT`, and various `COOLDOWN` states.
    -   **State Machine**: The loop iterates through the time series data bar-by-bar. If the strategy is in a position, it checks for exit conditions in the following order of priority: stop-loss, time-based exit, take-profit. If the strategy is flat and not in a cooldown period, it checks for entry signals.
    -   **Trade Logic**:
        -   **Entry**: An entry signal is triggered if the absolute z-score is above a threshold. The direction of the trade (long or short) is determined by the sign of the z-score.
        -   **Take-Profit (TP)**: A soft TP is triggered if both the z-score has reverted and the yield has moved favorably. A hard TP is triggered if the yield profit alone exceeds a hard threshold.
        -   **Stop-Loss (SL)**: A stop-loss is triggered if the position has lost more than a hard threshold in yield terms.
        -   **Time Exit**: A position is exited if it has been held for longer than `max_holding_bars`.
    -   **Execution Model**: The backtester simulates L2 order book execution by "walking the book". It calculates the volume-weighted average price (VWAP) for each leg of the trade. The execution is gated by liquidity and cost checks to ensure realism.
-   **`results.py`**: Defines the `MRBacktestResults` dataclass, which is a container for all the data generated by a backtest run, including per-bar data (positions, equity, PnL) and per-trade data (entry/exit bars, sizes, costs). It also provides convenient methods to convert the results to pandas DataFrames.
-   **`state.py`**: Defines the `ActionCode`, `State`, and `TradeType` enums, which are used throughout the backtester to represent different states and outcomes.

## 3. `sweep` Module

This module orchestrates the execution of multiple backtest scenarios.

-   **`sweep.py`**: The main entry point for running a sweep. It takes a list of scenarios and distributes them to different backends for execution (serial, thread, or process). It can run in "full" mode (returning all backtest results) or "metrics_only" mode (returning only a summary of the metrics).
-   **`sweep_runner.py`**: Provides a high-level, end-to-end interface for running sweeps directly from a configuration file.
-   **`multi_ref_runner.py` and `multi_ref_results_reader.py`**: Seem to be for handling sweeps with multiple reference instruments, but they are not fully implemented in the version I reviewed.
-   **`sweep_persist.py`**: Handles the saving of sweep results to disk, including metadata, summary tables, and full results for top-performing scenarios.
-   **`sweep_results_reader.py`**: The counterpart to `sweep_persist.py`, it loads the results of a sweep run from disk for post-hoc analysis.

## 4. `data` Module

-   **`data_loader.py`**: Defines the `BacktestDataLoader` class, which is responsible for loading market data from parquet files. It handles the alignment of data from different sources onto a common datetime index and pivots the data into the numpy arrays expected by the backtester.

## 5. `configs` Module

-   **`backtest_config.py`**: Defines the `MRBacktestConfig` dataclass, which holds all the scalar parameters for a single backtest run.
-   **`sweep_config.py`**: Handles the loading of sweep configurations from YAML files. A sweep configuration specifies a base configuration and a grid of parameters to sweep over.
-   **`utils.py`**: Contains helper functions for resolving config paths.

## 6. `parameters` Module

-   **`parameter.py`**: Defines the `MRParameter` enum. This enum lists all the tunable parameters of the backtesting model and associates a "preference direction" with each (+1 for more is better, -1 for less is better). This is used in the ranking of sweep results.

## 7. `analysis.py` and `plots.py`

-   **`analysis.py`**: Provides a rich set of functions for analyzing the results of a backtest run. It can compute a wide range of performance metrics (Sharpe ratio, Sortino ratio, max drawdown, etc.), pair entry and exit trades into round trips, and compute distributions of various statistics.
-   **`plots.py` (in `single_backtest` and `sweep`)**: Contains functions for generating various plots to visualize the backtest results, such as equity curves, drawdown plots, and PnL bar charts.

## Bug Review

I have reviewed the code for bugs and have not found any obvious logical errors. The code appears to be well-tested (based on the test files I have seen). The use of Numba for the core loop is a good optimization, and the code is written in a way that is compatible with it. The data handling and configuration management are robust. The separation of concerns is well-executed, which makes the code easier to understand and maintain.
